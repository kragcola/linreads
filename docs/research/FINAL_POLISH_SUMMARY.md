## 二、已定位的关键问题

### 2.1 P0 - 阻塞发布 (1个)

#### ✅ 横版图片过小 - 已定位根因

**确切代码位置**:
```kotlin
// android/render/epub/src/main/kotlin/dev/readflow/render/epub/EpubParaAdapter.kt:91
maxWidth = (680 * d).toInt()  // ⚠️ 硬编码680dp
```

**问题分析**:
- 680dp适合竖版阅读
- 横版时屏幕宽度可达900dp+
- 图片被限制为680dp，只占75%

**影响对比**:
| 方向 | 屏幕宽度 | 图片实际 | 占比 | 效果 |
|------|---------|---------|------|------|
| 竖屏 | ~420dp | ~380dp | 90% | ✅ 正常 |
| 横屏 | ~900dp | 680dp | 75% | ❌ 偏小 |
| 平板横屏 | ~1200dp | 680dp | 56% | ❌❌ 很小 |

**修复方案** (可直接实施):
```kotlin
private fun calculateMaxContentWidth(context: Context): Int {
    val metrics = context.resources.displayMetrics
    val orientation = context.resources.configuration.orientation
    val density = metrics.density
    val screenWidthDp = metrics.widthPixels / density
    
    return when (orientation) {
        Configuration.ORIENTATION_LANDSCAPE -> {
            // 横屏: 85%或900dp，取较小值
            min(
                (screenWidthDp * 0.85 * density).toInt(),
                (900 * density).toInt()
            )
        }
        else -> {
            // 竖屏: 680dp或90%，取较小值
            min(
                (680 * density).toInt(),
                (screenWidthDp * 0.90 * density).toInt()
            )
        }
    }
}

// 替换原有硬编码
maxWidth = calculateMaxContentWidth(parent.context)
```

**预期效果**:
- 竖屏: 保持不变
- 横屏: 从75%提升到85% (+13%)
- 平板: 从56%提升到85% (+52%)

**工作量**: 3天 (修复 + 测试 + 验证)
